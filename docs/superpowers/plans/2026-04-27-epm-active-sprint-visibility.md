# EPM Active Sprint Visibility Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the selected sprint value visible and changeable in the EPM Active view anywhere sprint state scopes EPM rollup results.

**Architecture:** Reuse the existing `renderSprintControl(surface)` in `frontend/src/dashboard.jsx` instead of creating a second sprint control. Render it only for EPM tabs where `shouldUseEpmSprint(epmTab)` is true, in both the main header and compact sticky header. Add source guards that fail when the EPM Active control surface no longer exposes the sprint control.

**Tech Stack:** React 19, esbuild, Node `node:test` source guards.

---

## Scope

In scope:

- Show the existing sprint dropdown in EPM Active mode.
- Show the selected sprint value in both main controls and compact sticky controls.
- Preserve current ENG sprint control behavior.
- Preserve EPM All/Completed behavior where sprint does not scope results.
- Add automated source guard coverage for EPM sprint control placement.
- Verify visually that EPM Active shows the selected sprint value.

Out of scope:

- No backend route changes.
- No EPM rollup query changes.
- No new sprint state or duplicate sprint picker component.
- No EPM settings changes.
- No OAuth branch changes.

## File Map

- `frontend/src/dashboard.jsx`: render `renderSprintControl('main')` and `renderSprintControl('compact')` inside EPM controls when `shouldUseEpmSprint(epmTab)` is true.
- `tests/test_epm_view_source_guards.js`: add source guards that fail if EPM Active controls do not expose the sprint control in main and compact headers.
- `frontend/dist/dashboard.js`, `frontend/dist/dashboard.js.map`: generated by `npm run build`.

## Plan Self-Review

- Spec coverage: The plan directly fixes the reported issue: EPM Active must show the selected sprint value. It includes both main and compact headers because both are active control surfaces.
- Backend scope: No backend or API changes are included because the bug is visual/control placement, not request construction.
- Regression protection: The guard fails if future extraction keeps passing `selectedSprint` to the backend but hides the sprint control again.
- Visual verification: Required because the missed requirement was user-facing visibility, not only data flow.
