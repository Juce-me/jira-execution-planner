# CSS Feature-Owned Partials Implementation Plan

> **Status:** Done. Executed and merged in [PR #75](https://github.com/Juce-me/jira-execution-planner/pull/75). Kept for audit context only.

> **For implementers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Split the remaining large dashboard CSS partials into feature-owned source files so targeted UI changes can be investigated by feature area without changing shipped CSS behavior.

**Architecture:** Keep `frontend/src/styles/dashboard.css` as the only CSS build entrypoint and keep one generated `frontend/dist/dashboard.css` artifact. Introduce nested style entrypoints by feature area, then move existing CSS blocks without selector rewrites, class renames, JSX changes, or cascade reordering beyond explicit ordered imports. Treat this as a source-organization change first; selector tightening belongs only in later feature work unless a selector must move with its complete override chain.

**Tech Stack:** Plain CSS partials bundled by esbuild, React 19 JSX class names, existing source-guard tests, Playwright UI specs, generated `frontend/dist` output from `npm run build`.

**Cross-Plan Order:** If this plan is executed with `DONE-planning-selection-card-grid.md`, run the Planning card-grid plan first, commit and verify it, then execute this CSS split. That keeps the Planning plan's current `eng.css`/`planning.css` file map accurate and lets this plan move the new Planning selectors into feature-owned partials.

---

## Scope And Design Contract

- Preserve the single shipped stylesheet: `/frontend/dist/dashboard.css`.
- Preserve the current JS build contract: `dashboard.jsx` may continue importing `./styles/dashboard.css`, but `build:js` still treats `.css` as empty and `build:css` owns emitted CSS.
- Do not convert styles to CSS-in-JS, CSS modules, inline style objects, or component JS imports in this slice.
- Do not rename class names, add React props, change JSX structure, or hand-edit `frontend/dist`.
- Keep import order explicit and stable. If moving selectors into subfolders, import those files from ordered feature entrypoints instead of relying on filesystem ordering.
- Move complete selector families together, including hover/focus states, media-query overrides, reduced-motion overrides, and keyframes that only serve that family.
- Keep shared primitives in a shared source file only when they are intentionally reused by multiple feature packages.
- Add source-structure guards so future CSS changes do not silently re-accumulate in oversized top-level partials.

## Validation Phase Anti-Creep Gates

During implementation review, reject the diff if any of these are true:

- Runtime CSS delivery changes from one bundled `frontend/dist/dashboard.css`.
- Component files are changed only to satisfy the CSS split.
- Selectors are renamed, broadened, or narrowed without a feature-specific reason and visual verification.
- `frontend/src/styles/eng.css`, `settings.css`, `stats.css`, `scenario.css`, `planning.css`, or `epm.css` remain large implementation files instead of becoming thin ordered entrypoints or compatibility imports.
- Existing tests are relaxed to look only at bundled `dist` when a source ownership assertion is still practical.
- Generated `frontend/dist/dashboard.css` is edited manually instead of produced by `npm run build`.

## Implementation Notes

- Execute this plan after `DONE-planning-selection-card-grid.md` when both plans are in scope. The Planning card-grid plan intentionally modifies the current `eng.css` and `planning.css`; this plan should then move the resulting Planning selectors into their feature-owned files.
